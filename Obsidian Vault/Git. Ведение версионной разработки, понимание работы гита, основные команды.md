
**Git** - распределенная система контроля версий. С её помощью разработчики фиксируют изменения в коде, просматривают историю проекта и синхронизируют репозитории

Репозиторий - это папка проекта вместе с историей изменений

Гит называется распределенной системой, потому что у каждого разработчика есть полноценная локальная копия репозитория. Поэтому с кодом и историей можно поработать на своем компьютере

Когда локальная работа завершена, коммиты можно передавать дальше. Git умеет обмениваться изменениями напрямую - например, через SSH. Но на практике чаще используют платформы вроде GitHub, GitLab или BitBucket: там удобнее работать вместе, управлять доступом и проводить код-ревью.

Важно не путать их с самим Git:

* Git - это инструмент, который управляет историей изменений
* GitHub, GitLab и подобные - платформы, где хранятся Git-репозитории и где команды совместно работают с кодом.

**Как Git хранит историю проекта**

История проекта в Git состоит из коммитов. Коммит фиксирует состояние файлов проекта в определенный момент времени и сохраняет его в истории репозитория.

При этом гит хранит историю не как список отдельных правок в файлах, а как снимки состояния проекта (snapshots). Каждый коммит - это снимок того, как выглядели файлы в момент сохранения. Когда нужно посмотреть разницу между версиями, гит сравнивает такие снимки и показывает изменения.

**Как Git отслеживает изменения и готовит их к коммиту** 

В работе с Git файлы могут находиться в трех состояниях:

* Modified - файл изменен в рабочей директории (working directory), но эти изменения еще не подготовлены к коммиту
* Staged - изменения подготовлены в промежуточную область, staging area (её также называют index), и попадут в следующий коммит.
* Committed - изменения уже сохранены в репозитории в виде коммита.

В итоге процесс выглядит так: меняются файлы в рабочей директории, изменения добавляются в staged area, а затем создается коммит.

	Staging area нужна, чтобы контролировать содержимое коммита. Например, можно изменить несколько файлов, но добавить в staging area только часть изменений - и в коммит попадут только они

Важно понимать, что Git не копирует все файлы заново при каждом коммите. Если файл не изменился, Git просто использует уже существующую версию и ссылается на неё. Это экономит место и в целом ускоряет работу.

**Основные команды Git**

| **Команда**                     | **Что делает**                                                                       |
| ------------------------------- | ------------------------------------------------------------------------------------ |
| `git --version`                 | Проверяет, установлен ли Git и показывает его версию                                 |
| `cd путь/к/папке`               | Переходит в нужную директорию в терминале                                            |
| `git init`                      | Создаёт новый Git-репозиторий в текущей папке                                        |
| `git clone <адрес_репозитория>` | Клонирует удалённый репозиторий и создаёт его локальную копию                        |
| `git status`                    | Показывает состояние репозитория: изменённые, подготовленные и неотслеживаемые файлы |
| `git diff`                      | Показывает разницу между текущими изменениями и последним коммитом                   |
| `git diff --staged`             | Показывает изменения, которые уже добавлены в staging area                           |
| `git log`                       | Показывает историю коммитов                                                          |
| `git log --oneline`             | Показывает историю коммитов в кратком формате                                        |
| `git add <файл>`                | Добавляет файл в staging area для следующего коммита                                 |
| `git add .`                     | Добавляет все изменённые и новые файлы в staging area                                |
| `git commit -m "сообщение"`     | Создаёт коммит и сохраняет подготовленные изменения                                  |
| `git commit --amend`            | Изменяет последний коммит (сообщение или состав файлов)                              |
| `git branch`                    | Показывает список веток                                                              |
| `git branch <имя_ветки>`        | Создаёт новую ветку                                                                  |
| `git switch <имя_ветки>`        | Переключается на указанную ветку                                                     |
| `git switch -c <имя_ветки>`     | Создаёт новую ветку и сразу переключается на неё                                     |
| `git merge <ветка>`             | Объединяет указанную ветку с текущей                                                 |
| `git rebase <ветка>`            | Переносит коммиты текущей ветки поверх указанной ветки                               |
| `git push`                      | Отправляет локальные коммиты в удалённый репозиторий                                 |
| `git push -u origin <ветка>`    | Отправляет ветку в удалённый репозиторий и связывает её с удалённой веткой           |
| `git fetch`                     | Скачивает изменения из удалённого репозитория без применения                         |
| `git pull`                      | Скачивает изменения из удалённого репозитория и объединяет их с текущей веткой       |
| `git restore --staged <файл>`   | Убирает файл из staging area                                                         |
| `git checkout <commit>`         | Переключается на конкретный коммит (состояние detached HEAD)                         |
